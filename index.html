🔧 它怎么工作

上传后，文档被切成带页码的小块；提问时检索出最相关的几块作为「来源」，再让模型据此回答并标注 [1][2]。
PDF→ 切块→ 向量/BM25→ 检索 Top-K→ 带引用回答

设置 / API Key ×

🔒 你的 Key 只存在本机浏览器（localStorage），请求直连模型厂商，不经过任何中间服务器。本工具是一个纯静态网页。

RAG 是怎么"带来源回答"的 ×

1
解析 & 切块用 pdf.js 在浏览器里抽出文本，切成 ~600 字、带页码的小块（保留重叠避免切断语义）。
2
向量化有 Key 时，每个块经嵌入模型变成一个向量；问题也变成向量。无 Key 时改用 BM25 关键词打分（纯前端，支持中英文）。
3
检索 Top-K按余弦相似度 / BM25 分数，挑出和问题最相关的几块——这就是"来源片段"。
4
带引用生成把这几块拼进提示词，要求模型只依据它们回答并标注 [1][2]，点引用就能跳回原文。找不到就如实说"文档未提及"。
这正是 Open WebUI 等工具内部 RAG 的最小骨架。区别只在工程规模：向量库、混合检索、OCR、多文档等，都是在这四步上加厚。
